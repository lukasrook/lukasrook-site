Kill the 404s: Hosting a Project Site on GitHub Pages

Today’s lesson: GitHub Pages project sites live under /repo-name/. If your links assume the root domain, you’ll get a face full of 404s—especially on mobile.

The bug

I launched a home page at lukasrook.github.io/lukasrook-site/ and added a blog overview under /blog/. On desktop it worked (because I was testing locally), but on production every link to /blog/... pointed to https://lukasrook.github.io/blog/ which doesn’t exist. Classic mismatch between local paths and the Pages base URL.

The fix

  1. Confirmed the real URLs: https://lukasrook.github.io/lukasrook-site/blog/ and .../blog/day-001.html.
  2. Replaced every hard-coded /blog/… reference with /lukasrook-site/blog/… (nav, buttons, footer, cards).
  3. Committed & pushed so GitHub Pages rebuilt. A quick curl -I confirmed HTTP 200 for the new URLs.
  4. Hard-refresh on mobile to flush the cached 404 page.

Takeaway

If you deploy a project site on GitHub Pages, either:

  • Reference assets with the {{ site.baseurl }} variable (if you’re using Jekyll), or
  • Manually prefix every internal link with /repo-name/ to match the hosted path.

Skip that step and your visitors will bounce from a dead page.
